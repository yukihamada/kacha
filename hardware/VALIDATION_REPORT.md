## 1. 🔴 CRITICAL: CR2032バックアップ回路の設計欠陥

### 問題
設計書に「CR2032でWiFi通知を1回送信できる（約30秒動作）」と記載あるが、**物理的に不可能**。

### 計算

```
CR2032 スペック:
  容量:         225 mAh
  内部抵抗:     10〜30 Ω (新品時10Ω、劣化時30Ω)
  最大連続放電:  ~3 mA 推奨
  パルス最大:    ~15 mA

ESP32-S3 WiFi送信時:
  ピーク電流:   300 mA
  平均電流:     120 mA (WiFi接続 + MQTT送信 2秒間)

CR2032 @ 300mA: 電圧降下 = 300mA × 10Ω = 3V
→ 端子電圧 = 3.0V - 3.0V = 0V ← 完全に電圧崩壊
→ 1N5819Wの順方向電圧(0.3V)を含めると更に不利
```

**結論**: CR2032単体でWiFi送信は不可能。電圧が崩壊してMCUリセットが発生する。

### 修正案

**Option A (推奨): 大容量コンデンサをパラ接続**
```
CR2032 --- 1N5819W --- ┬ --- 3.3V Rail
                       │
                    470μF 電解コンデンサ (追加)
                       │
                      GND
```
- WiFiピーク電流(300mA)は470μFが瞬時供給
- CR2032は補充電流(3mA)として機能
- コスト追加: ~$0.10 (LCSC C1572)

**Option B: BLEのみで緊急通知**
- WiFiの代わりにBLE Advertisingで通知
- BLE TX: ~15mA → CR2032で数十回送信可能
- ファームウェア側でWiFi切断時はBLE通知に切り替える

**→ Option A + B 両方実装を推奨**

## 2. 🔴 CRITICAL: リセットボタンのGPIOアサイン矛盾

### 問題
2箇所で記述が矛盾している:

| 場所 | 記述 |
|------|------|
| schematic_notes.md セクション2 GPIO表 | `BUTTON_RESET → GPIO0` |
| schematic_notes.md セクション5 ボタン説明 | `リセットボタンはENピンに接続` |

### 正解

ESP32-S3のブート動作:
- **GPIO0 = LOW at boot → Download Mode（ファームウェア書き込みモード）**
- リセットボタンをGPIO0に接続すると、ボタンを押しながら電源投入でDLモードに入るため誤動作する

**正しい設計:**
```
リセット (SW1): ENピン ── SW1 ── GND (正しい。ENをLOWにすると確実にリセット)
書き込み用:     GPIO0 をテストパッドに引き出しておく (JTagやフラッシュツール用)
```

### 修正
BOM/Schematic修正:
- SW1: EN → GND (変更なし、ただしGPIO表の記述を修正)
- GPIO0: テストパッドとして基板に露出 (製造後のファームウェア書き込み用)

## 3. ⚠️ MAJOR: I2Cプルアップ抵抗値の再検討

### 現状
R1, R2 = 10KΩ @ 3.3V で 5デバイス共有バス

### 検証計算

```
I2C Fast Mode (400kHz) の立ち上がり時間要件: t_r < 300ns (Standard) / 100ns (Fast)

推定バス容量:
  ESP32-S3内部: ~5pF
  SHT45:        ~5pF
  BH1750:       ~5pF
  SGP40:        ~5pF
  LIS2DH12:     ~5pF
  BMP280:       ~5pF
  配線 (55mm PCB, 2本): ~20pF
  合計:          ~50pF (十分マージンあり、上限400pF)

立ち上がり時定数: τ = R × C = 10,000 × 50×10⁻¹² = 500ns
0→0.7Vdd (立ち上がり) に要する時間: τ × ln(1/0.3) ≈ 600ns

→ 400kHzのt_r要件(300ns)を超えている可能性あり
```

**推奨**: 10KΩ → **4.7KΩ** に変更
- 4.7KΩ: τ = 235ns → 立ち上がり ~280ns ✅
- 電流増加: (3.3V / 4.7KΩ) × 2 = 1.4mA 追加 (許容範囲)

**または**: 400kHzではなく100kHz(Standard mode)で動作させれば10KΩでも問題なし

## 4. ⚠️ MAJOR: AMS1117熱設計

### 計算

```
最大入力電流条件:
  ESP32-S3 WiFi TX ピーク: 300mA
  センサー合計:             ~10mA
  WS2812B (フル輝度):       60mA
  合計ピーク:               370mA

LDO消費電力 = (Vin - Vout) × Iload
= (5.0V - 3.3V) × 0.37A = 0.629W

AMS1117 SOT-223 熱抵抗: θJA = 100°C/W (銅箔なし), 50°C/W (銅箔あり)

Tj(max) @ 25°C ambient:
  銅箔なし: 25 + (0.629 × 100) = 87.9°C  → OK (limit: 125°C)
  銅箔あり: 25 + (0.629 × 50)  = 56.5°C  → 余裕あり
```

**推奨**: AMS1117のGNDパッド周辺に最低 **300mm²の銅箔ベタ** を配置すること。
これがないと長時間運用でLDOが熱保護に入り電圧が不安定になる。

## 6. ⚠️ MAJOR: Rustファームウェアの"no_std"記述

### 問題
DESIGN.mdに「Rust (ESP-IDF, no_std対応)」と記載があるが:

- **WiFi + MQTT over TLS を no_std で実装するのは極めて困難**
- ESP-IDF の WiFi スタック (lwip, mbedTLS) は std 環境前提
- Embassy + embassy-net を使えば no_std でも WiFi 可能だが、ESP32-S3向けサポートはまだ実験的

### 推奨
```toml
# Cargo.toml
[dependencies]
esp-idf-hal = "0.43"      # std環境, FreeRTOS上で動作
esp-idf-svc = "0.48"      # WiFi, MQTT, HTTP
embedded-hal = "1.0"
```

**`no_std` → `std (ESP-IDF HAL)` に変更する**。
機能的には全て実現可能で、開発効率が大幅に向上する。
