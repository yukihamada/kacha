## 1. デバイス層 (ESP32-S3)

### 1.1 FreeRTOSタスク構成

| # | タスク名 | 優先度 | スタック | 周期 | 役割 |
|---|---------|--------|---------|------|------|
| 1 | `safety_monitor` | 5 (最高) | 8KB | 10秒 | Safety Ladder状態機械・ACS計算・Tier判定 |
| 2 | `mmwave_reader` | 4 | 4KB | UART割込+10秒 | LD2410B UART詳細データ取得・OUTピン監視 |
| 3 | `sensor_poll` | 3 | 4KB | 60秒 | SHT40 I2C読み取り (Lite) / 全I2Cセンサー (Hub) |
| 4 | `cloud_sync` | 2 | 8KB | 60秒+イベント | MQTT送信・監査ログアップロード・コマンド受信 |
| 5 | `led_control` | 1 | 2KB | 100ms | WS2812B制御・状態表示パターン |
| 6 | `ota_updater` | 1 | 8KB | オンデマンド | ATECC608A署名検証 + OTAダウンロード・適用 |

**GPIO割り込み (タスク外)**:
- `GPIO3` (ドアセンサー): ANYEDGE割り込み → `safety_monitor` に通知
- `GPIO2` (I'm OKボタン): NEGEDGE割り込み → `safety_monitor` に通知
- `GPIO1` (VBUS ADC): 停電検知 → BLEフォールバック起動

**優先度設計の根拠**:
- safety_monitor が最高優先度: 人命に直結するタスクは絶対にスケジュール遅延しない
- cloud_sync は低め: ネットワーク遅延を吸収してもミッションに影響しない
- led_control は最低: 表示の遅延は許容できる

### 1.2 センサー読み取りスケジュール (省電力最適化)

```
時間軸 →
0s       10s      20s      30s      40s      50s      60s
│        │        │        │        │        │        │
├─mmWave─┤─mmWave─┤─mmWave─┤─mmWave─┤─mmWave─┤─mmWave─┤  10秒間隔
│                                                      │
├──────────────────── SHT40 ──────────────────────────┤  60秒間隔
│                                                      │
├──────────────────── ATECC608A (署名) ───────────────┤  イベント時のみ
│                                                      │
├──────────────────── MQTT送信 ───────────────────────┤  60秒間隔
```

**省電力戦略**:
- 正常時: WiFiはMODEM_SLEEP (DTIM beacon間隔でスリープ)
- 深夜 (睡眠推定時): mmWave読み取りを30秒間隔に延長
- ACS > 0.8 かつ 深夜: クラウド送信を5分間隔に延長
- 停電時: WiFi OFF → BLE Advertise only → Deep Sleep 30秒サイクル

### 1.4 停電時フォールバック (CR2032 + BLE only)

```
USB断検知 (GPIO1 ADC < 1.0V)
  │
  ├→ 即座: WiFi OFF (省電力)
  ├→ 即座: mmWave OFF (5V依存のため動作不可)
  ├→ BLE Advertise開始
  │   └→ Service UUID: 0xFE95 (KAGI Safety)
  │   └→ Characteristic: 停電フラグ + 最後のACS + タイムスタンプ
  │   └→ 家族のスマホが近くにあれば直接受信
  │
  ├→ 30秒後: Deep Sleep (30秒間隔で起床)
  │   └→ 起床時: BLE Advertise 5秒 → Deep Sleep
  │   └→ OKボタン割り込みで即起床
  │
  └→ CR2032容量: 225mAh
     BLE Advertise: 平均15mA x 5秒 = 75mAs/サイクル
     Deep Sleep: 10uA x 30秒 = 0.3mAs/サイクル
     1サイクル = 75.3mAs ≈ 0.021mAh
     持続: 225mAh / 0.021mAh ≈ 10,700サイクル ≈ 89時間
     → 約8時間 (安全係数10倍) のBLE通知が可能
```

## 4. セキュリティモデル

### 4.1 ATECC608A 信頼チェーン

```
製造時 (JLCPCB工場)
  │
  ├→ ATECC608A スロット構成:
  │   Slot 0: KAGI Root CA 公開鍵 (読取専用、書込ロック)
  │   Slot 1: デバイス秘密鍵 (生成後読取不可)
  │   Slot 2: デバイス証明書 (Root CAが署名)
  │   Slot 3: OTA署名検証用公開鍵
  │   Slot 4-7: 将来予約
  │
  └→ 信頼チェーン:
     KAGI Root CA (HSM保管)
       └→ デバイス証明書 (Slot 2)
            └→ MQTT mTLS クライアント認証
            └→ 監査ログ署名
            └→ OTA署名検証

検証フロー:
  デバイス → クラウド: mTLS (デバイス証明書で認証)
  クラウド → デバイス: OTA署名 (Root CA公開鍵で検証)
  デバイス → ログ: ECDSA署名 (デバイス秘密鍵で署名)
  誰でも → ログ検証: デバイス公開鍵で署名を検証可能
```

### 4.2 デバイスプロビジョニングフロー

```
Step 1: 工場プロビジョニング (製造ライン)
  ├→ ATECC608A に秘密鍵を生成させる (チップ外に出ない)
  ├→ 公開鍵を取得 → CSR生成 → Root CAで証明書発行
  ├→ 証明書をSlot 2に書き込み
  ├→ OTA公開鍵をSlot 3に書き込み
  ├→ Slot 0,1 を永久ロック
  └→ device_id をNVSに書き込み

Step 2: ユーザーセットアップ (SoftAP)
  ├→ ESP32-S3 が "KAGI-SETUP-XXXX" AP起動
  ├→ ユーザーがスマホで接続 → WebUIでWiFi SSID/PW入力
  ├→ デバイスがWiFi接続テスト → 成功
  ├→ クラウドにデバイス登録 (mTLSで認証)
  └→ 家族アカウントとデバイスを紐付け

Step 3: 運用中の鍵管理
  ├→ 秘密鍵: ATECC608A内で永久保護 (物理攻撃にも耐性)
  ├→ 証明書: 有効期限5年。更新はOTA経由
  └→ Root CA: HSM (AWS CloudHSM or YubiHSM) で保管
```

### 4.3 鍵ローテーション

```
デバイス証明書 (5年有効):
  有効期限90日前 → クラウドからリニューアル通知 (MQTT)
  デバイス: 新CSR生成 (ATECC608Aの別スロット)
  クラウド: Root CAで新証明書発行 → デバイスに送信
  デバイス: 新証明書でmTLS接続テスト → 成功 → 旧証明書を無効化

OTA署名鍵 (10年有効):
  ローテーション時は新旧両方の公開鍵で検証可能な期間を設ける
  新公開鍵はOTA更新で配布 → 旧鍵での署名も6ヶ月間有効

Root CA (20年有効):
  ローテーション不要。万が一の場合は全デバイスの工場リセット。
  → だからこそRoot CA鍵はHSMで厳重に保管する
```

## 5. 障害モード分析 (FMEA)

### 5.1 WiFi断絶 → BLEフォールバック

```
検知: cloud_syncタスクが MQTT PINGREQ の応答を3回連続失敗
  │
  ├→ WiFi再接続試行 (exponential backoff: 5s, 10s, 20s, 40s, 80s)
  │
  ├→ 5分間復旧しない場合:
  │   ├→ LED: 青点滅 (通信障害)
  │   ├→ BLE Advertise開始 (停電時と同じプロトコル)
  │   ├→ ローカルログバッファに記録 (最大100件リングバッファ)
  │   └→ Safety Ladderはローカルで継続動作
  │       ├→ Tier1: LED+ブザーで本人に通知 (正常動作)
  │       ├→ Tier2: BLE経由で近くの家族スマホに通知
  │       └→ Tier3: 不可 (クラウド必須) → Tier2で停止
  │
  └→ WiFi復旧時:
      ├→ ローカルバッファの全ログをクラウドに送信
      ├→ タイムスタンプ付きで遡及記録
      └→ LED: 緑 (復旧)
```

### 5.2 クラウド障害 → ローカルアラート継続

```
影響範囲:
  - MQTT送信不可 → ローカルバッファに蓄積
  - プッシュ通知不可 → BLEフォールバック
  - OTA不可 → 現行FWで安全に動作継続

デバイス側の動作:
  - Safety Ladderは100%ローカル実行 (クラウド非依存)
  - ACS計算・Tier判定・LED/ブザー制御はクラウドなしで動作
  - OKボタン・ドアセンサー・mmWaveは全てローカル処理
  → 「クラウドが死んでも人は守れる」設計

クラウド復旧時:
  - MQTT再接続 → バッファ一括送信
  - 家族アプリに「一時的に接続が切れていましたが復旧しました」通知
  - 監査ログの整合性チェック (タイムスタンプ順にソート)
```

### 5.3 センサー故障 → 縮退動作

```
センサー故障検知方法:
  - I2Cデバイス: ACK応答なし → 故障フラグ
  - LD2410B: UART応答タイムアウト(5秒) → 故障フラグ
  - ドアセンサー: 検知不能 (常にHIGH or LOW固定) → 故障フラグ

縮退モード:
  ┌───────────────────────────────────────────────────────┐
  │ 故障センサー         │ 対応                            │
  ├───────────────────────────────────────────────────────┤
  │ LD2410B (mmWave)    │ ACSからmmWave項を除外。           │
  │                     │ ドア+ボタンの2センサーで判定      │
  │                     │ Tier1閾値を30分短縮 (安全寄り)   │
  │                     │ LED: 橙点滅 (センサー異常)        │
  │                     │ クラウドに故障報告                │
  ├───────────────────────────────────────────────────────┤
  │ SHT40 (温湿度)      │ 温度項をACSから除外               │
  │                     │ 安否確認機能への影響は軽微         │
  ├───────────────────────────────────────────────────────┤
  │ ドアセンサー          │ ACSからドア項を除外              │
  │                     │ mmWave+ボタンの2センサーで判定    │
  ├───────────────────────────────────────────────────────┤
  │ ATECC608A           │ 署名なしでログ記録 (※要注意)      │
  │                     │ クラウドに緊急故障報告            │
  │                     │ OTA更新を一時停止 (署名検証不可)  │
  ├───────────────────────────────────────────────────────┤
  │ 全センサー故障        │ LED赤常灯 + ブザー連続           │
  │                     │ BLEで「デバイス故障」通知          │
  │                     │ クラウドに緊急報告                │
  └───────────────────────────────────────────────────────┘
```

### 5.4 停電 → 8時間BLE only

```
前提: CR2032 (225mAh) + C5 (470uF バースト吸収)

電力バジェット:
  BLE Advertise (5秒/30秒サイクル): 平均 ~0.4mA
  Deep Sleep (25秒/30秒サイクル): ~0.01mA
  加重平均消費: ~0.07mA

  理論持続時間: 225mAh / 0.07mA ≈ 3,214時間 (133日)
  安全係数適用 (CR2032の内部抵抗・温度・自己放電):
    実効容量: ~100mAh (低温時の最悪ケース)
    実効持続: 100 / 0.07 ≈ 1,428時間 (59日)
    → 8時間は十分なマージンで達成可能

動作シーケンス:
  T+0s:    停電検知 → WiFi/mmWave OFF → BLE ON
  T+5s:    BLE Advertise (停電通知 + 最後のACS)
  T+30s:   Deep Sleep
  T+60s:   Wake → BLE Advertise → Deep Sleep (以降繰り返し)
  T+5min:  OKボタンが押されれば BLE Advertise で「OK」送信
  T+8h:    まだ動作中 (バッテリー余裕あり)
  T+24h+:  電圧低下 → BLE停止 → 完全停止
           → 復電時にNVSから最後の状態を復元
```
