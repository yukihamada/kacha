## 設計原則

1. **偽陰性ゼロ** -- 「大丈夫」と判断して実は危険、は許さない
2. **偽陽性の最小化** -- 「狼少年」になると家族が反応しなくなる
3. **プライバシー優先** -- 音声・映像は記録しない。存在/非存在のみ
4. **段階的エスカレーション** -- 本人確認 → 家族 → 緊急連絡先 の順
5. **ハードウェア署名** -- ATECC608Aで全イベントに署名。改ざん不可能

## センサーフュージョン戦略

複数センサーの重み付き統合で「生存確信度 (Alive Confidence)」を算出する。

```
┌────────────────────────────────────────────────────────┐
│ 生存確信度 (Alive Confidence Score)                       │
│                                                          │
│   ACS = Σ(w_i * s_i) / Σ(w_i)                          │
│                                                          │
│   w_i = センサーiの重み, s_i = センサーiの生存シグナル(0-1) │
├──────────────┬─────────────────────────────────────────┤
│ センサー       │ シグナル s_i           │ 重み w_i │ 時定数 │
├──────────────┼───────────────────────┼──────────┼───────┤
│ LD2410B      │ 呼吸検知=1.0           │ 5.0      │ 3h    │
│              │ 動体検知=0.8           │          │       │
│              │ 検知なし=0.0           │          │       │
├──────────────┼───────────────────────┼──────────┼───────┤
│ I'm OKボタン  │ 押した=1.0 (減衰あり)   │ 4.0      │ 24h   │
├──────────────┼───────────────────────┼──────────┼───────┤
│ ドアセンサー   │ 開閉あり=0.8           │ 3.0      │ 24h   │
├──────────────┼───────────────────────┼──────────┼───────┤
│ 温湿度 (SHT40)│ 20-28°C=0.3(生活環境) │ 1.0      │ --    │
├──────────────┼───────────────────────┼──────────┼───────┤
│ 音 (Hub以上)  │ 生活音あり=0.6         │ 2.0      │ 4h    │
├──────────────┼───────────────────────┼──────────┼───────┤
│ CO2 (Pro)    │ 400-1500ppm=0.5       │ 2.0      │ --    │
│              │ (呼気による上昇)       │          │       │
└──────────────┴───────────────────────┴──────────┴───────┘

ACS閾値:
  ACS >= 0.6  → 正常 (緑LED)
  0.3 <= ACS < 0.6 → 注意 (黄LED, 監視強化)
  ACS < 0.3  → Tier1発動
```

### 時間減衰関数

各センサーのシグナルは最後の検知時刻から時間減衰する:

```
s_i(t) = s_i_raw * exp(-lambda_i * (t - t_last))

lambda_i = ln(2) / T_half_i

  T_half_i: センサーiの半減期
    LD2410B:    T_half = 90分  (3時間で0.25に減衰)
    I'm OKボタン: T_half = 720分 (24時間で0.06に減衰)
    ドアセンサー:  T_half = 720分
    音:          T_half = 120分 (4時間で0.25)
```

**例**: LD2410Bが最後に呼吸を検知してから3時間経過:
```
s_mmwave(180min) = 1.0 * exp(-0.0077 * 180)
                 = 1.0 * exp(-1.386)
                 = 0.25

ACS(Liteモデル) = (5.0*0.25 + 4.0*0.5 + 3.0*0.4) / (5.0+4.0+3.0)
               = (1.25 + 2.0 + 1.2) / 12.0
               = 0.37  → 「注意」ゾーン
```

## 生活リズム学習アルゴリズム (ベイズ推定)

### 概要

最初の7日間のデータから個人の「正常パターン」を学習し、
閾値を自動調整する。夜型の人に昼型の基準を適用しない。

### 確率モデル

各時間帯 h (0-23時) におけるセンサーイベントの発生率をポアソン過程でモデル化:

```
X_h ~ Poisson(lambda_h)

  X_h: 時間帯hでのイベント数 (mmWave検知, ドア開閉, ボタン押下の合計)
  lambda_h: 時間帯hの平均イベント発生率

事前分布: lambda_h ~ Gamma(alpha_0, beta_0)
  alpha_0 = 2.0  (弱い事前分布)
  beta_0 = 1.0

学習期間のデータ (n日分) から事後分布を更新:
  lambda_h | data ~ Gamma(alpha_0 + Σx_h, beta_0 + n)

事後期待値: E[lambda_h] = (alpha_0 + Σx_h) / (beta_0 + n)
```

### 異常検知

学習済みの lambda_h を用いて、現在の時間帯での「活動なし」確率を計算:

```
P(X_h = 0 | lambda_h) = exp(-lambda_h)

異常スコア A_h = -log(P(X_h = 0 | lambda_h)) = lambda_h

例: 午前10時の平均イベント率 lambda_10 = 8.5 (活発な時間帯)
    この時間に活動なし: A_10 = 8.5 → 異常度「高」

例: 午前3時の平均イベント率 lambda_3 = 0.2 (睡眠中)
    この時間に活動なし: A_3 = 0.2 → 異常度「低」(正常な睡眠)
```

### 閾値の適応的調整

```rust
// firmware/src/safety/rhythm.rs

struct DailyPattern {
    // ベイズ事後分布パラメータ (時間帯別)
    alpha: [f32; 24],   // Gamma事後パラメータ alpha
    beta: [f32; 24],    // Gamma事後パラメータ beta
    // 要約統計量
    wake_time: (u8, u8),    // 起床時刻の推定 (中央値)
    sleep_time: (u8, u8),   // 就寝時刻の推定
    max_door_interval: u32, // ドアが開く最長間隔 (分)
    typical_quiet_duration: u32, // 通常の静止時間 (分)
    days_learned: u16,      // 学習日数
}

impl DailyPattern {
    /// 新しいデータで事後分布を更新 (オンライン学習)
    fn update(&mut self, hour: u8, event_count: u32) {
        self.alpha[hour as usize] += event_count as f32;
        self.beta[hour as usize] += 1.0;
        self.days_learned += 1;
    }

    /// 現在時刻での異常スコアを計算
    fn anomaly_score(&self, hour: u8, minutes_since_last_event: u32) -> f32 {
        let h = hour as usize;
        let lambda = self.alpha[h] / self.beta[h];

        // 活動がない時間が長いほど異常度が上がる
        // P(silence >= t) = exp(-lambda * t/60)
        let t_hours = minutes_since_last_event as f32 / 60.0;
        lambda * t_hours
    }

    /// 適応的閾値: 個人の生活リズムに基づく安全係数
    fn adaptive_threshold(&self) -> SafetyThresholds {
        // 安全係数: 学習データが少ないうちは保守的 (2.0)、
        // データが増えるにつれて個人化 (1.5)
        let safety_factor = if self.days_learned < 7 {
            2.0
        } else if self.days_learned < 30 {
            1.5 + 0.5 * (1.0 - self.days_learned as f32 / 30.0)
        } else {
            1.5
        };

        SafetyThresholds {
            no_motion_alert: (self.typical_quiet_duration as f32
                             * safety_factor) as u32,
            door_closed_alert: (self.max_door_interval as f32
                               * safety_factor) as u32,
        }
    }
}
```

### 曜日別パターン

平日と休日でパターンが異なる人 (出勤者) への対応:

```
lambda_{h,d} ~ Gamma(alpha_{h,d}, beta_{h,d})

d = 0 (平日) or 1 (休日)

最初の14日間 (平日10日+休日4日) で分離学習開始。
それ以前は曜日区別なしで統合モデルを使用。
```

## 誤報率の計算根拠

### モデル

```
偽陽性率 (FPR) = P(Tier1発動 | 実際は安全)

単一センサーの場合:
  FPR_single = P(ACS < 0.3 | alive) ≈ 誤検知率

センサーフュージョンの場合 (独立性仮定):
  FPR_fused ≈ Π(FPR_i) (各センサーが同時に誤検知する確率)
```

### 各センサーの誤検知率

| センサー | 個別誤検知率 | 根拠 |
|---------|------------|------|
| LD2410B (呼吸不検知) | 2% | Hi-Link公称: 98%検知率 @3m |
| ドアセンサー (24h未開) | 5% | 引きこもり日・体調不良日 |
| I'm OKボタン (24h未押) | 10% | 押し忘れ |
| 音 (4h無音) | 15% | 静かな読書・昼寝 |

### Liteモデルの誤報率計算

Tier1は ACS < 0.3 で発動。3センサー (mmWave, button, door) が全て減衰した場合:

```
P(全センサーが同時に誤検知) = P(mmWave誤) * P(button誤) * P(door誤)
                          = 0.02 * 0.10 * 0.05
                          = 0.0001 = 0.01%

1日あたりの判定回数: 1440回 (1分ごと)
1日あたりの期待偽陽性回数: 1440 * 0.0001 = 0.144回

→ 月あたり: 0.144 * 30 = 4.3回

ただしTier1は30分の応答待ちがある:
  真の偽陽性 = Tier1発動 × ボタン押し忘れ率
            = 4.3 * 0.10 = 0.43回/月
```

### 目標と改善戦略

| 指標 | 初期 (学習期間なし) | 学習後 (30日) | 目標 (運用1年後) |
|------|-------------------|-------------|----------------|
| Tier1発動 (誤報含む) | ~5回/月 | ~2回/月 | < 1回/月 |
| Tier2到達 (誤報) | ~0.5回/月 | ~0.2回/月 | < 0.1回/月 |
| Tier3到達 (誤報) | 0 (opt-in前提) | 0 | 0 |

**誤報削減のフィードバックループ**:
```
Tier1発動 → 本人が [問題なし] → 匿名統計収集 (opt-in)
  → 「この時間帯は正常な長時間静止」として学習
  → lambda_h の更新 → 閾値が個人に最適化
  → 次回以降の誤報が減る
```

## 呼吸検知の技術的限界

### LD2410B (60GHz FMCW) の物理的特性

```
送信周波数: 60-64GHz (Vバンド)
変調方式:   FMCW (周波数変調連続波)
波長:       ~5mm
帯域幅:     4GHz → 距離分解能 3.75cm
最大検知距離: 6m (存在検知) / 6m (動体検知)
検知角度:    水平約60度 (コーン形状)
```

### 呼吸検知の限界ケース

| ケース | 検知可否 | 理由 | 対策 |
|--------|---------|------|------|
| 仰向けで寝ている | 検知可能 | 胸の上下動をキャッチ | -- |
| 横向きで寝ている | 検知可能 | 腹部の動きをキャッチ | -- |
| うつ伏せで寝ている | やや困難 | 背面の動きが小さい | ドア+ボタンで補完 |
| 厚い布団の中 | 精度低下 (-10%) | 布団の減衰 | 季節補正パラメータ |
| 壁の裏側 | 木造: 可能 / RC: 不可 | 壁材による減衰 | 同室設置を推奨 |
| 複数人在室 | 最も近い人を検知 | 反射強度による | 民泊では問題なし |
| ペット (猫・小型犬) | 検知することがある | 呼吸パターン類似 | ペットモード設定 |

### 重要な設計判断

LD2410Bは「100%の呼吸検知」を保証するものではない。
しかし、PIRの「0%の静止人体検知」と比較すれば劇的な改善。
**不完全でも、「ないよりはるかにまし」な安全ネットを、$2で実現する。**

不足分はドアセンサーとI'm OKボタンで補完する。
この3層構造 (mmWave + ドア + ボタン) がKAGI Liteの設計根拠。

## 「I'm OK」ボタン -- 最重要UIの設計

### なぜボタンが最重要か

mmWaveレーダーには限界がある (前節参照)。ドアセンサーも完璧ではない。
しかし**本人がボタンを押す行為**は、確実な生存証明になる。

**テクノロジーの限界を、最も原始的なインタラクションで補完する。**

### ボタンのUX要件

```
物理仕様:
  サイズ: 12mm径以上 (高齢者の指でも押しやすい)
  ストローク: 0.5mm以上 (押した感触が確実に伝わる)
  力: 1.5N以下 (関節リウマチの人でも押せる)
  色: 緑色のリング (「安心」のシグナル)

設置場所の推奨:
  玄関の靴箱の上 / リビングのリモコン置き場 / 寝室の枕元
  → 毎日の動線上に自然に存在する場所

フィードバック:
  押した瞬間: ピッ (短い確認音) + LED緑フラッシュ
  結果: スマホアプリに「お父さんがOKを押しました 8:32」
  家族の安心: 何もなければ何もない。それが最高のUX。
```

### 習慣化の仕組み

問題: 高齢者がボタンを毎日押す習慣をどう作るか?

```
1日目-7日目 (学習期間):

8日目以降:
  リマインド: 個人の起床時刻 + 30分に調整
  24時間押さない: Tier1開始 (ただし他センサーで正常なら延期)

30日後:
  ほとんどの利用者が習慣化 (行動心理学: 21日で習慣形成)
  リマインド音を徐々に小さくしていく

理想状態:
  「朝起きたらKAGIのボタンを押す」が
  「朝起きたら顔を洗う」と同レベルの日課になる
```

## 既知の誤報ケースと対策

| ケース | 誤報の理由 | 対策 |
|--------|----------|------|
| 旅行中 | 長期不在 | アプリで「旅行モード」設定 (最大30日) |
| 病院入院 | 長期不在 | 同上 |
| 夜間トイレ後すぐ戻る | ドア開閉でリセット | ドア開閉でACSリフレッシュ |
| 読書・テレビ長時間 | mmWave検知中 | 呼吸検知がACSを維持 → 誤報にならない |
| センサー方向に障害物 | mmWave遮断 | 設置時に必ず検知テスト実施 |
| 停電 | WiFi切断 | CR2032+BLE経由でスマホ直接通知 |
| WiFi障害 | 接続断 | 15分以内に復帰しなければアラート |
| ペットの動き | mmWaveが反応 | ペットモードで閾値調整 |
| 季節変動 | 冬の布団 | 季節パラメータで感度補正 |
