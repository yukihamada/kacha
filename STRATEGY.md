# Enabler — プロダクト戦略

## 思想: ローカルファースト × オープンソース

### なぜローカルファーストか

大手（freee, Money Forward, CloudSign）は全データをクラウドに預ける。
ユーザーは自分のデータの主権を失い、月額を払い続けなければデータにアクセスできない。

**Enablerの立場:**
- データはユーザーの端末に。サーバーには暗号化データか何も置かない
- アプリを消してもKeychainに残る。乗り換え自由
- オープンソースだからコードを検証できる。バックドアがないことを証明できる
- 「あなたのデータは、あなたのもの」

これは思想であり、差別化であり、マーケティングメッセージでもある。

### なぜオープンソースか

1. **信頼**: セキュリティを謳うアプリがクローズドソースなのは矛盾。見せられるから信頼される
2. **コミュニティ**: 開発者が使って、貢献して、広めてくれる
3. **持続性**: 会社が潰れてもコードは残る。ユーザーは安心して使える
4. **採用**: 「このコードを書いた」が最高のポートフォリオ

### ビジネスモデル（オープンソース + 収益化）

| 層 | 内容 | 価格 |
|---|---|---|
| Free (OSS) | 全機能。ローカルデータ。広告なし | 無料 |
| Pro | E2Eシェア（サーバー利用分）、バックグラウンド同期、チーム管理 | ¥980/月 |
| Business | 複数物件、Beds24連携、収支レポート、優先サポート | ¥2,980/月 |

サーバーコストがかかる機能だけ有料。ローカル機能は全て無料。

## 名前の結論

### シリーズ化しない。個別ブランドで勝負する

理由:
- 5つのアプリを「セット」で訴求しても、ユーザーは1つずつしかダウンロードしない
- 「パシャファミリー」は開発者の都合。ユーザーには関係ない
- 各アプリが独立して検索・評価される方がASO的に強い
- 成功したアプリから他を知ってもらえばいい（クロスプロモ）

### 推奨名

| 現名 | 新名 | 理由 |
|---|---|---|
| カチャ | **KAGI (カギ)** | 鍵=家の象徴。グローバルで通じる。4文字。App Store検索で「鍵 管理」に引っかかる |
| パシャ | **パシャ** (変更なし) | 撮影音として認知度あり。「パシャ レシート」で検索される可能性 |
| チャリン | **チャリン** (変更なし) | お金の音は直感的。サブタイトルで「請求書」を明示 |
| ポン | **ポン** (変更なし) | 判子の音。電子署名と親和性高い |
| サクッ | **サクッ** (変更なし) | 確定申告をサクッと。動詞的で覚えやすい |

**カチャだけ変更する理由:**
- カチャは他4つと客層が違う（物件管理 vs フリーランスツール）
- 「カチャ」は何のアプリか全く想像できない
- 「KAGI」なら鍵→家→スマートホーム→物件管理と連想が働く
- kagi.app はブラウザ（検索エンジン）だが、App Storeでは競合しない

## ローカルファースト vs クラウド: 結論

| | ローカルファースト | クラウド |
|---|---|---|
| データ主権 | ユーザーにある | サービス側にある |
| プライバシー | 最強 | 漏洩リスク |
| オフライン | 完全動作 | 依存 |
| サーバーコスト | 最小（シェア機能のみ） | 高い |
| 複数端末同期 | 手動（Keychainバックアップ） | 自動 |
| データ移行 | 容易 | ロックイン |
| ビジネスモデル | 機能課金 | データ人質 |

**結論: ローカルファーストが正解。**

理由:
1. サーバーコストが低い → 無料版を永続的に提供できる
2. データ漏洩リスクがゼロに近い → セキュリティが最大の差別化
3. freee/MFとの真逆のポジション → ニッチだが強固
4. オープンソースとの相性が最高 → コミュニティが育つ
5. 長期的に「あなたのデータは、あなたのもの」が勝つ

複数端末同期が必要になったら、iCloud CloudKitを使えばAppleのインフラでE2E同期できる。自前サーバー不要。
